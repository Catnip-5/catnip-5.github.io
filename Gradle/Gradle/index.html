Wrapper 包装器

作用

  • 版本控制： 通过 gradlew（Linux/Mac）或 gradlew.bat（Windows）脚本，项目可以锁定特定 Gradle 版本，无需手动安装 Gradle。
  • 环境无关： 开发者无需在本地安装 Gradle，Wrapper 会自动下载指定的 Gradle 版本。
  • 一致性： 保证团队成员和 CI/CD 环境使用相同的 Gradle 版本，避免构建问题。

主要文件

  • 位于项目根目录的 gradle/wrapper/ 文件夹中
    • gradlew Linux/Mac 的可执行脚本。
    • gradlew.bat Windows 的批处理脚本。
    • gradle-wrapper.jar Wrapper 的核心逻辑，负责下载和运行 Gradle。
    • gradle-wrapper.properties 配置文件，指定 Gradle 版本和下载地址。
  • gradle-wrapper.properties 文件

build.gradle 文件

plugins 插件

插件作用

  • 自动化任务： 插件可以定义构建、测试、打包等任务。
  • 约定配置： 提供默认设置，例如 Java 项目的目录结构。
  • 扩展功能： 支持特定技术栈，如 Android、Spring、Docker 等。

插件类型

  • 脚本插件
  • 二进制插件

声明插件

  • 使用 plugins 块（推荐，声明式语法）
  • 使用 apply 方法（传统方式）
  • 应用脚本插件

  • 中央仓库

dependencies 依赖

依赖类型

  • implementation 用于项目的实现代码，编译和运行时需要。依赖不会暴露给其他模块，减少传递性依赖冲突
  • api 用于库项目的公共 API，编译和运行时需要。依赖会暴露给使用该库的其他模块（传递性）
  • compileOnly 仅在编译时需要，运行时不需要（例如注解处理器）
  • runtimeOnly 仅在运行时需要，编译时不需要（例如数据库驱动）
  • testImplementation 用于测试代码的编译和运行（如 JUnit）
  • testCompileOnly / testRuntimeOnly 类似 compileOnly 和 runtimeOnly，但仅用于测试

依赖方式

  • 外部库依赖 从 Maven Central、JCenter 或其他仓库（如 Google）下载的库
  • 本地文件依赖 使用本地 JAR 文件或目录
  • 项目依赖 依赖同一多模块项目中的其他模块

动态版本

  • 固定版本： 如 1.2.3。
  • 动态版本： 使用 +（如 1.2.+）或范围（如 [1.0,2.0)）。
